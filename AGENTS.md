# AGENTS.md

## 1. Purpose
이 문서는 이 저장소에서 Codex와 개발자가 **일관된 기준으로 안전하게 변경**하기 위한 고정 작업 규칙이다.

- 변경은 항상 **작동 보존, 범위 최소화, 책임 분리**를 우선한다.
- 구현 선택은 취향이 아니라 **명확한 근거(변경 용이성, 테스트 용이성, 복잡도 통제)**로 결정한다.
- 이 문서는 레포의 현재 사실(Project Facts)과, 재사용 가능한 엔지니어링 원칙을 분리해 제공한다.

---

## 2. Project Facts
아래는 현재 레포에서 확인된 사실만 정리한다.

- 프론트엔드 앱은 **JavaScript + React 18** 기반이다.
- 빌드/개발 도구는 **Vite 5 + @vitejs/plugin-react**다.
- 패키지 설정은 `"type": "module"`이며, ESM `import/export`를 사용한다.
- SPA 엔트리포인트는 `src/main.jsx`이고, `App`을 `#root`에 마운트한다.
- 전역 상태 라이브러리(Redux, Zustand 등)는 사용하지 않고, React `useState/useMemo/useCallback` 및 커스텀 훅으로 상태를 관리한다.
- 런타임 영속 값은 커스텀 훅 `useLocalStorage`를 통해 `localStorage`에 저장한다.
- 네트워크 호출은 브라우저 내장 **fetch** 기반이며, `src/api/client.js`의 `apiClient`로 중앙화되어 있다.
- 서버 인증은 `Authorization: Bearer <token>` 헤더를 사용한다.
- 폴링은 `useJobPolling` 훅에서 `setInterval`로 처리한다.
- 정규화/파싱 유틸(`normalizeBaseUrl`, `parseListInput`)이 분리되어 있다.
- `legacy-cli/` 아래에 별도 Node 기반 레거시 CLI 코드가 있으며, 프론트 `src/`와 분리되어 있다.

## 3. Core Engineering Philosophy

### 3.1 FM하고 정석적인 설계
- 규칙 없는 빠른 구현보다, 경계와 책임이 보이는 정석 설계를 선택한다.
- 이유: 유지보수 시 의사결정 비용을 줄이고, 변경 리스크를 낮춘다.

### 3.2 OOP + FP 혼합
- 객체는 책임과 상태 전이를 담당한다.
- 함수는 변환과 조합을 담당한다.
- 이유: 도메인 의미(객체)와 데이터 흐름(함수)을 분리하면 복잡도가 통제된다.

### 3.3 단순성 우선
- 처음 구현은 가장 단순한 형태로 시작한다.
- 필요가 증명될 때만 확장한다.
- 이유: 과잉 설계를 방지하고 변경 속도를 유지한다.

### 3.4 합성 중심 성장
- 기능은 작은 단위(도메인 규칙/유틸/훅/컴포넌트)의 합성으로 확장한다.
- 이유: 부분 교체와 재사용이 쉬워진다.

### 3.5 책임 중심 설계(SRP)
- 객체/함수/컴포넌트는 한 가지 변화 이유를 갖도록 유지한다.
- 이유: 수정 영향 범위를 예측 가능하게 만든다.

### 3.6 UI와 비즈니스 규칙 분리
- UI는 표현/이벤트 연결에 집중하고, 비즈니스 규칙은 도메인·유스케이스로 분리한다.
- 이유: UI 변경과 정책 변경을 독립적으로 다룰 수 있다.

### 3.7 부수효과는 가장자리로 격리
- 네트워크, 스토리지, 타이머, 브라우저 API 접근은 boundary 계층으로 모은다.
- 이유: 테스트 가능성과 디버깅 가능성이 높아진다.

### 3.8 변경의 축 기준 추상화
- 추상화는 “실제로 자주 바뀌는 축”에서만 도입한다.
- 이유: 미래 예측 기반의 불필요한 추상화를 방지한다.

### 3.9 계층 경계 존중
- Domain / Usecase / Adapter / View 경계를 명확히 지킨다.
- 이유: 의존 방향이 안정되면 기능 추가 시 파급이 줄어든다.

## 4. Architecture Guidance
- 도메인 객체와 도메인 규칙은 UI 컴포넌트와 분리한다.
- 컴포넌트는 표현과 사용자 상호작용 처리에 집중한다.
- 변환/정규화/파싱은 순수 함수로 분리한다.
- 네트워크 호출은 API client/adapter 계층으로 모은다.
- `localStorage`, network, timer 등 부수효과는 hook/service/adapter로 모은다.
- 비즈니스 규칙을 컴포넌트 body 전반에 흩뿌리지 않는다.
- 훅은 오케스트레이션, 유틸은 순수 변환, 컴포넌트는 렌더링/이벤트 연결을 담당한다.

---

## 5. State and Data Rules
- 상태는 현재 레포가 사용하는 방식(React state + 훅)과 일관되게 다룬다.
- 전역 상태 라이브러리가 없는 영역에는 임의로 새 상태 솔루션을 도입하지 않는다.
- 새 상태 관리 도구 도입은 성능·복잡도·공유 범위 문제를 근거로 명시적으로 제안할 때만 허용한다.
- 원시 객체/배열을 컴포넌트에서 직접 변이하지 않는다.
- 상태 전이는 명시적 함수/유스케이스/도메인 메서드로 표현한다.
- 데이터 정규화·검증·변환은 별도 함수로 분리한다.

---

## 6. Networking and IO Rules
- 네트워크 호출은 중앙 client/adapter에 모은다.
- 컴포넌트에서 URL 조합, 헤더 조합, 에러 파싱 로직을 반복 작성하지 않는다.
- 외부 I/O는 boundary에서 처리한다.
- 도메인 계층은 브라우저 API 및 네트워크 세부 구현을 알지 않도록 유지한다.
- 현재 레포 facts 기준 네트워크는 fetch를 사용하므로, 신규 호출도 기본적으로 동일 패턴을 따른다.

---

## 7. Code Style Rules (Hard Rules)

### 7.1 No else (Non-negotiable)
- `else`는 절대 사용하지 않는다.
- `else if`도 사용하지 않는다.
- 분기는 guard clause / early return으로 작성한다.
- 실패/예외/빈 값/무효 입력을 먼저 반환한다.
- 중첩 `if`는 평탄화한다.
- 삼항 연산자는 허용하되, 중첩 삼항은 금지한다.
- 신규 코드, 리팩토링 코드, 예시 코드, 문서 예제 모두 동일하게 적용한다.
- Codex는 결과물에 `else`가 남지 않았는지 자체 점검한다.

### 7.2 Guard clause 우선
- 함수 시작부에서 실패 조건, 예외 조건, nullish 조건을 먼저 처리한다.
- happy path는 아래로 자연스럽게 흐르게 작성한다.

### 7.3 Const-first
- 기본은 `const`를 사용한다.
- 재할당이 꼭 필요할 때만 `let`을 사용한다.
- `var`는 금지한다.

### 7.4 No magic numbers / No magic strings
- 의미 있는 숫자/문자열 리터럴은 명명된 상수로 승격한다.
- timeout, interval, retry, limit, key, label 등은 상수로 관리한다.

### 7.5 Params rule
- 파라미터가 3개 이상이면 객체 인자를 우선 검토한다.
- 핵심 식별자 1~2개는 유지하고 부가 옵션은 객체로 묶는다.

### 7.6 Naming
- 이름은 역할과 책임이 드러나야 한다.
- `util`, `data`, `handler`, `temp` 같은 모호한 이름 남용을 금지한다.

### 7.7 Nullish-safe iteration
- 배열 순회/메서드 사용 시 nullish-safe 패턴을 선호한다.
- 예: `(items ?? []).map(...)`

## 8. OOP and FP Usage Rule
- OOP는 책임 있는 객체, 상태 전이, 도메인 규칙, 응집된 행위에 사용한다.
- FP는 변환, 정규화, 매핑, 필터링, 조합, 파이프라인, 파생 계산에 사용한다.
- 도메인 객체는 가능한 명시적인 행위를 제공한다.
- 순수 계산 로직은 순수 함수로 유지한다.
- 객체와 함수의 역할이 불명확하게 섞이지 않도록 경계를 유지한다.
- 팀 기본 원칙: **객체가 책임을 수행하고, 함수가 조합을 돕는다.**

---

## 9. JSDoc Rules
- public 함수, 도메인 메서드, 유스케이스, 복잡한 유틸에는 JSDoc을 작성한다.
- 한국어로 짧고 명료하게 작성한다.
- “무엇을 하는지”와 “언제 쓰는지”를 드러낸다.
- 구현을 줄 단위로 반복 설명하는 주석은 금지한다.

---

## 10. React Frontend Guidance
- 컴포넌트는 작고 읽기 쉽게 유지한다.
- 큰 컴포넌트는 패널/폼/리스트/상세 등 역할 기준으로 분리한다.
- 훅은 재사용 가능한 상태, 동기화, 오케스트레이션 로직에 사용한다.
- 렌더링 코드와 데이터 준비 코드를 분리한다.
- 폼 파싱, 목록 정규화, 에러 메시지 변환은 util/usecase로 분리한다.
- 뷰는 thin 하게 유지한다.

---

## 11. Change Safety Rules
- 기존 레포 구조와 진입점을 임의로 변경하지 않는다.
- 새 라이브러리 도입은 반드시 필요할 때만 한다.
- 대규모 리팩토링보다 국소적 변경을 우선한다.
- 먼저 기존 패턴을 따르고, 명확한 문제 증거가 있을 때만 새 패턴을 도입한다.
- “지금 필요한 변경”과 “미래를 위한 추상화”를 구분한다.

---

## 12. Output Style for Codex
- 항상 최소 변경으로 문제를 해결한다.
- 관련 없는 파일은 수정하지 않는다.
- 새 파일 추가 시 필요성을 설명 가능해야 한다.
- 리팩토링 커밋과 동작 변경 커밋은 분리하는 것을 원칙으로 한다.
